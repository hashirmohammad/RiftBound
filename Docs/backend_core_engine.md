###Goal 
The goal of this task was to implement the core backend simulation layer for Riftbound.
This system manages the full game state, player state, turn system, card runtime behavior,
and rune mechanics so that the UI can interact with the game through structured actions instead
of directly modifying data.

###What Was Built
The backend simulation architecture was implemented with the following components:
action — Represents a player’s intended action during the game.
card_data — Static card template loaded from JSON containing card stats and metadata.
card_database — Responsible for loading and managing card template data from the card JSON file.
card_instance — Runtime representation of a card during a match including UID, zone, and state.
game_engine — Controls game setup, turn progression, action processing, and core game rules.
game_state — Stores the full match state including players, phases, turn number, and event log.
player_state — Manages player zones including deck, hand, board, graveyard, rune deck, and rune pool.
player_turn — Manages the turn system and phase progression of the game.
rune_instance — Defines the RuneInstance class, which represents a rune during gameplay.

###How to Test
Step 1 — Run the debug scene
	- Open the backend debug scene
	- Run the scene (F6)

Expected behavior:
	- Game initializes correctly
	- Rune decks are built and shuffled
	- Runes channel during the CHANNEL phase
	- Cards can be played from hand
	- Event logs print turn progression and actions

###Important Notes for the Team
- The backend uses UIDs generated by GameState.next_uid() to uniquely identify runtime objects such as cards and runes.
- The frontend should interact with the backend only through GameAction objects.
- Card templates (CardData) are separated from runtime objects (CardInstance) to keep game data clean and scalable.
- The rune system now uses runtime rune objects instead of placeholder values, allowing future mechanics like rune selection, rune types, and advanced costs.
